## 1. Purpose

Translate the behavioural rules in `behaviouralDescription.md` and the strategy decisions in `operatorObservationStrategy.md` into a numbered list of functional requirements that an implementer can build against and a reviewer can test against.

This document is **functional**: it describes *what* the system must do, observable to a user or an administrator. **Every implementation choice** (technology stack, file layouts, data structures, URL patterns, cookie formats, dialects, libraries) lives in the companion **Technical Design Specification**, `technicalDesignSpecification.md`. Each TDS item back-references the FR IDs it serves; see [TDS §11].

Each requirement carries:

- a stable **ID** (e.g. `FR-REC-040`) for traceability,
- a single **"shall" statement** describing observable behaviour,
- short **notes** (rationale, source clauses, TDS pointers),
- **acceptance criteria** — a SMART pass/fail check that determines whether the requirement is met,
- a **rollout band** — Step 1 (the core recorder) or Step 2 (polish) per strategy §6,
- a **MoSCoW priority** — `M` Must, `S` Should, `C` Could, `W` Won't (this release).

### 1.1 SMART acceptance criteria

Every requirement's **Acceptance criteria** column is written to be SMART, so the suite of criteria taken together constitutes an executable acceptance test for the system:

| SMART trait | How this FDS applies it |
|---|---|
| **S — Specific** | A single observable behaviour or property. No "the system should be fast" — instead "GET / with a valid cookie returns the home page within 500 ms". |
| **M — Measurable** | An objective pass/fail check. Numbers wherever timing, counts, or sizes matter; explicit response codes; byte-for-byte equality where applicable. |
| **A — Achievable** | Realistic on the stack pinned in [TDS §3] (PHP 7.4+, SQLite, generic shared hosting) at the small-deployment scale assumed in §6 (≤ 100 users, ≤ 1 000 observations, ≤ 100 photos). Timing budgets are deliberately generous so cheap shared hosting can meet them under realistic load. |
| **R — Relevant** | The criterion tests *this* FR — not a side effect. If an AC requires functionality from a different FR, it cross-references that FR. |
| **T — Time-bound** | Timing is named explicitly: "within 500 ms", "after 24 h", "every request", "within 30 s after network restore". Non-timing ACs name the trigger event ("after admin clicks delete..."). |

A test suite that exercises every AC and passes constitutes acceptance of the corresponding FRs. ACs are written for the *observable* behaviour; tests may legitimately implement them via UI automation, HTTP probes, direct DB inspection, or whatever combination matches the system under test.

The rollout band answers *when does this ship*; the MoSCoW priority answers *how critical is it*. The two are complementary: a Step 2 item can still be a **Must** if its absence would make the Step 2 release pointless, and a Step 1 item can be a **Should** if the Step 1 release is still useful without it. Items in §8 are the **Won't** set for this release — they have no FR ID by design.

Requirements without an explicit rollout band are Step 1.

## 3. Definitions

| Term | Meaning |
|---|---|
| **User / Operator** | A person in or around the greenhouse who records observations on a phone. Self-asserted identity, no password. |
| **Admin / Administrator** | A person who configures the app, manages users, and exports data. Identified by a name supplied from the configuration; authenticated by a password set via a first-run setup wizard (FR-SEC-020) and stored hashed in persistent storage. |
| **Greenhouse** | One physical installation. The system supports more than one from day zero [OS §10.2]. Each has a stable greenhouse ID and a friendly name. |
| **Observation** | One recorded event: timestamp + operator + category + tag + optional severity + optional note + optional photo. |
| **Category** | One of the top-level groupings the operator picks first. The launch taxonomy has five categories (Wellbeing, Environment, Crop, Sensor/Control, Maintenance) [OS §3]; three climate-impact categories (Temperature, Humidity — too high, Humidity — too low) were added via `migrations/0002_climate_taxonomy.sql`, bringing the active total to **8**. |
| **Tag** | One of ≤ 6 sub-choices under a category. Identified by a stable internal ID, displayed via a separate display name. |
| **Cookie** | The single browser cookie that binds a phone to a user record. The only credential a user has. |
| **Edit window** | The interval after recording during which the operator can still modify or delete their own observation. Default 24 h [OS §4]. |
| **Admin URL** | A non-discoverable path that opens the admin GUI [BD §28]. The default value and configurability are specified in [TDS §7]. |

## 4. Actors

| Actor | Channel | How they are identified |
|---|---|---|
| Operator | Mobile-optimised user GUI | A single browser cookie that the server maps to the user record [BD §12, §18]. |
| Administrator | Desktop-suitable admin GUI at a separate URL | Admin name + password from the application's configuration; a server-side session that times out [BD §27]. |
| Analyst (read-only, off-system) | CSV export | Not authenticated by the app — they receive CSV files produced by an administrator [OS §2 "merge"]. |

The analyst is not a runtime actor — they consume exported CSVs and are listed only for context.

| A6 | The app targets **modern evergreen browsers**: the latest two major versions of Chrome, Safari (iOS + macOS), Firefox, and Edge. ES2017 baseline JS, no transpilation. Larger or longer-running deployments need the budgets re-evaluated. | operator scope brief |

### 6.5 Greenhouse context and administration (`FR-GH`)

| ID | Requirement | Notes | Acceptance criteria | Step | MoSCoW |
|---|---|---|---|---|---|
| FR-GH-010 | The system shall support multiple greenhouses from launch, each with a stable greenhouse ID and a friendly name. The greenhouse ID shall match **`^[0-9A-F]{4}$`** (exactly 4 **uppercase** hexadecimal characters). By convention this is the hex representation of the **last 2 bytes of the paired greenhouse-controller's MAC address** (e.g. controller MAC `AA:BB:CC:DD:5E:3F` → greenhouse ID `5E3F`), which makes IDs deterministic and meaningful to operators who know their controller. Deployments without a paired controller may assign any unique 4-hex value. Dialect choice favours GDPR portability over Excel convenience; a user who wants the Excel variant can re-encode the file themselves. | From the user settings page, clicking "Download mijn gegevens" produces a download within **30 s** for ≤ **1 000** observations. The archive contains: an observations file (rows = the user's observations only, RFC-4180 CSV per [TDS §9] — `,` separator, UTF-8 no BOM), an account text file (handle, creation timestamp, current/last greenhouse, last-seen timestamp), and any photos referenced by the user's observations preserving the date-shard layout from [TDS §4]. | 2 | M |
| FR-SEC-060 | The administrator shall be able to configure a **data-retention duration** (FR-INST-030 key, default **365 days**). When set to > 0 days, the app's periodic sweep automatically deletes: (a) observations whose `ts` is older than `now() − retention_days`, together with any attached photos; (b) `admin_audit` rows older than a **fixed 90-day** window — audit value decays faster than observation value, so the audit log is pruned more aggressively. With retention set to `0`, no automatic deletion occurs for either category. | Bounds the database against unlimited growth. The 365-day default keeps roughly a year of observation context, matching the seasonal cycle of greenhouse cultivation. Enforcement mechanism and sweep cadence: [TDS §4]. Pairs with FR-USR-060 for orphan-user cleanup. | With `retention_days = N` (N > 0) and seed observations dated `ts < now() − N days`, the next sweep deletes them and their photo files within **24 h** of crossing the threshold. Seed `admin_audit` rows dated > 90 days ago are also deleted on every sweep when `retention_days > 0`. With `retention_days = 0`, **neither** observations **nor** audit rows are auto-deleted regardless of age. | 2 | S |

## 8. Out of scope

For symmetry with [OS §8], requirements that are explicitly **not** in this document and **not** to be implemented:

- Any communication between the observation app and the greenhouse controller.
- Any live-data display (current temperature, window state, etc.).
- Any control surface affecting the controller.
- Server-side push notifications, email, or SMS to operators.
- Multi-tenant isolation beyond the greenhouse-ID partitioning described in §6.5 (all greenhouses share one admin and one database).
- HTTPS termination, certificate issuance, and firewall configuration — these are the administrator's responsibility per A1.
